Activity的启动模式

  1. standard

    ApplicationContext去启动standard模式的Activity时会 提示错误

    因为非Activity类型的Context不具有任务栈，但是standard模式启动的Activity是需要放入任务栈中，所以我们可以添加FLAG_ACTIVITY_NEW_TASK标记为，让他创建一个新的任务栈即可

  2. singleTop

    如果活动已经位于栈顶，那么只会调用onPause->onNewIntent->onResume

    不处于栈顶那么就会新建一个活动

  3. singleTask

    栈中存在便弹出上方的所有活动；不存在将会判断其所需任务栈，然后进行创建（参见理解启动模式和taskAffinity之间的关联）

  4. singleInstance

    单独创建一个任务栈

  5. singleInstancePerTask

设置启动模式的方法

  1. AndroidMenifest中指定android:LaunchMode
  2. 通过Intent设置标志位（如:FLAG_ACTIVITY_NEW_TASK）

优先级：第二种高于第一种

限定范围：第一种无法直接为Activity设置FKAG_ACTIVITY_TOP标识，第二种无法指定singleInstance模式

几个主要的Flags

  • FLAG_ACTIVITY_NEW_TASK 指定singleTask模式
  • FLAG_ACTIVITY_SINGLE_TOP 指定singleTop模式
  • FLAG_ACTIVITY_CLEAR_TOP 该Activity同一任务栈上的所有Activity均出栈
  • FLAG_ACTIVITY_EXCLUDE_FROM_RECENTS Activity不会出现在历史Activity列表中（等同于AndroidMenifest文件中的excludeFromRecents = “true”）

Intent匹配规则

启动Activity分为两种：显示调用和隐式调用

当两种调用同时存在时，以显示调用为主

隐式调用则是设置IntentFilter，其中过滤信息有action、category、data，需要同时满足过滤列表中的3种信息，过滤列表种的action、category、data可以有多个，只有完全匹配才可以启动目标Activity。

action：一个intent-filter中可以有多个action，匹配其中一个即可

category：可以不指定（系统会默认加上android.intent.category.DEAFULT，如果清单文件中不包含这个标志，那么无法接收隐式Intent），如果有多个category，那么每一个都需要与过滤规则匹配

data：需要匹配过滤规则中的任意data规则。清单文件中创建的data过滤规则的URI存在默认scheme（默认为content和file）。还需要注意Intent指定完整的data，必须调用setDataAndType，不可以先setData再setTyep，因为这两个方法会互相清除对方的值

注意：一个Activity中可以有多个intent-filter

检查匹配的Activity

  1. PackageManager的resolveActivity方法
  2. Intent的resolveActivity方法
  3. PackageManager的queryIntentActivity方法，该方法和上面不同的是，该方法返回所有匹配成功的Activity信息

IPC机制

IPC机制是指进程间通信或者跨进程通信，也是两个进程之间进行数据交换的过程。

多进程模式

开启多进程模式

只有一种方法，给四大组件在AndroidMenifest中指定android:precess属性

其实还有非常规方法：通过JNI在native层fork一个新的进程

查看：可以通过adb shell ps 或者 adb shelle ps | grep 包名 查看

两种进行命名方式

  1. :进程名 如:remote，这是属于当前应用的私有进程，其他应用的组件不可以和它跑在同一个进程中

  2. 包名.进程名 如com.example.android.myprocess 全局进程，其他应用可以通过ShareUID的方式和它跑在同一个进程中

    需要注意：需要具有相同的ShareUID和相同的签名才可以跑在同一进程

    不管是否在同一个进程中：此时它们可以互相访问私有数据，如data目录、组件信息等

    同一个进程中：还可以共享内存数据，看起来就像是一个应用的两个部分

多进程模式的运行机制

使用多进程会造成如下几方面的问题：

  1. 静态成员和单例模式完全失效

  2. 线程同步机制完全失效

  3. SharedPreferences的可靠性下降

  4. Application会多次创建

    通过在自定义的application中打印日志可以发现，即使是同一应用中，但是属于不同进程的Activity都会执行一次Application的创建，日志中出现的进程名也都不相同

虽然有多种问题，但是系统提供了很多种跨进程通信的方法

  1. Intent传递数据
  2. 共享文件
  3. SharedPreferences
  4. 基于Binder的Messenger
  5. AIDL
  6. Socket

IPC基础概念

包含3块内容

  1. Serializable接口
  2. Parcelable接口
  3. Binder

Serizlizable接口

该接口用于实现对象序列化，一般情况下是需要指定一个serialVersionUID，但是也可以不指定。

如果不指定，在类成员变量的数量、类型发生变化后，是无法正常反序列化的，因为系统会自动去生成类的serialVersionUID。

如果指定，可以很大程度上避免反序列化过程的失败，比如删除某个成员变量后，反序列化操作也依然可以成功。但是，如果类结构发生了非常规性改变，比如修改了类名、成员变量类型，即使serialVersionUID验证通过，反序列化过程依然会失败。

注意：静态成员变量属于类不属于对象，不会参与序列化过程；其次，用transient关键字标记的成员变量不参与序列化过程

我们可以重写writeObject和readObject方法改变系统默认的序列化过程

Parcelable接口

实现该接口后需要实现3个方法和添加一个静态对象

  • writeObject
  • describeContents
  • 构造方法(parcel:Parcel)
  • 静态类：Parcelable.Creator类型 该静态类需要实现两个方法
    • createFromParcel
    • newArray

Binder

Binder是Android中的一个类，实现了IBinder接口。

从IPC角度来说，Binder是Android中的一种跨进程通信方式；

从Android应用层来说，Binder是客户端和服务端进行通信的媒介，当bindService时，服务端会返回一个包含了服务端业务调用的Binder对象，通过这个Binder对象，客户端就可以获取服务端提供的服务或者数据，这里的服务包括普通服务和基于AIDL的服务

Android开发中，BInder主要用在Service中（包括AIDL和Messenger），普通Service中的Binder不涉及进程间通信。而Messenger的底层其实就是AIDL，所以下面使用AIDL来分析BInder的工作机制。
